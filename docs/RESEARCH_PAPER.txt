ABSTRACT

The increasing sophistication of automated cyber attacks demands network defense systems capable of autonomous detection, decision-making, and enforcement without human intervention. Traditional Intrusion Detection Systems (IDS) such as Snort [1] and Suricata [2] generate alerts but lack automated response capabilities, leaving a critical gap between threat detection and containment. This paper presents NTTH (No Time To Hack), an agent-inspired autonomous network defense architecture that implements a closed-loop pipeline from packet capture through threat scoring, enforcement, and attacker intelligence gathering. The system employs a hybrid risk scoring model combining signature-based IDS rules (weighted 0.6) with an unsupervised Isolation Forest [3] anomaly detector (weighted 0.4), validated through systematic ablation study across 11 weight configurations. Upon threat detection, the system autonomously applies Linux kernel-level nftables firewall rules and performs flow-aware dynamic redirection of specific attacker-victim-port flows to SSH and HTTP honeypots, capturing attacker commands, credentials, and behavioral patterns. We evaluate NTTH on a real-world dataset of 10,247 packets captured during controlled attack scenarios and cross-validate on the CICIDS2017 benchmark dataset [4]. Experimental results demonstrate a detection rate of 96.2% with a false positive rate of 3.8%, achieving a mean response latency of 127 milliseconds from packet capture to firewall rule enforcement. Comparative evaluation against Snort and Suricata shows that NTTH uniquely provides automated containment while maintaining comparable detection accuracy. The system is deployable on commodity hardware consisting of an Ubuntu laptop and a $10 Atheros AR9271 USB WiFi adapter, making enterprise-grade autonomous defense accessible for small-scale network environments.

I. INTRODUCTION

The landscape of cyber threats has evolved dramatically, with automated scanning tools, botnets, and increasingly, AI-driven attack agents capable of exploiting network vulnerabilities at machine speed [5]. According to the IBM X-Force Threat Intelligence Index [6], the average time from initial compromise to lateral movement has decreased to under two hours, while the average time for manual incident response remains 15 to 30 minutes for well-staffed Security Operations Centers (SOCs) and significantly longer for organizations with limited resources. This disparity between attack speed and human response capability creates an urgent need for autonomous defense systems that can detect, decide, and enforce without human intervention.

Traditional Intrusion Detection Systems (IDS) such as Snort [1] and Suricata [2] have been the cornerstone of network threat detection for over two decades. These systems excel at signature-based detection of known attack patterns but suffer from two fundamental limitations. First, they generate alerts without taking enforcement actions, requiring human operators to interpret alerts, correlate events, and manually configure firewall rules. Second, they cannot adapt to novel attack patterns that do not match existing signatures without manual rule updates. Studies have shown that the average organization receives over 10,000 IDS alerts per day, with analysts able to investigate only a fraction [7].

Honeypot systems such as Cowrie [8] and T-Pot [9] provide valuable attacker intelligence by simulating vulnerable services and capturing adversary behavior. However, existing honeypot deployments are predominantly static, requiring manual configuration and pre-deployment before attacks occur. There is no existing system that dynamically deploys honeypot redirections in response to real-time threat assessments.

Machine learning approaches to intrusion detection have gained significant traction, with models such as Random Forest [10], Support Vector Machines [11], and deep learning architectures [12] achieving high detection accuracy on benchmark datasets. However, these approaches typically function as classification engines disconnected from enforcement mechanisms, producing predictions that still require human interpretation and action.

Recent work on autonomous cyber defense has explored reinforcement learning for optimal response strategy selection (AARF [13]) and AI-generated deception assets (AETHER [14]), but these systems operate in cloud environments, do not measure response latency, and are not designed for commodity hardware deployment. The LLM Agent Honeypot [15] addresses the emerging threat of AI-powered attackers but provides detection only without enforcement capabilities.

This paper addresses these gaps by presenting NTTH, an agent-inspired autonomous network defense architecture that provides a complete closed-loop from packet capture to attacker command capture. Our key contributions are as follows:

    1) We design and implement an agent-inspired modular pipeline consisting of four autonomous processing stages (Threat Assessment, Decision Making, Enforcement, and Reporting) coordinated through an asynchronous event bus, enabling sub-second response latency of approximately 127 ms from packet capture to firewall rule enforcement.

    2) We propose a hybrid risk scoring model that combines signature-based IDS rules with an Isolation Forest [3] anomaly detector using empirically optimized weights determined through systematic ablation study, achieving 96.2% detection rate with 3.8% false positive rate.

    3) We implement flow-aware dynamic honeypot deployment that redirects specific attacker-victim-port flows to deception services while preserving the attacker's legitimate traffic to other destinations, a capability absent from existing static honeypot deployments [8][9].

    4) We integrate wireless threat monitoring using a commodity Atheros AR9271 USB adapter in monitor mode for WiFi probe request analysis [16], deauthentication attack detection, and device presence tracking, enabling convergent wired and wireless threat detection.

    5) We provide comprehensive experimental validation on both a real-world dataset of 10,247 packets and the CICIDS2017 benchmark [4], with comparative evaluation against Snort [1] and Suricata [2] demonstrating that NTTH uniquely provides automated containment while maintaining comparable detection accuracy.

The remainder of this paper is organized as follows. Section II reviews related work in intrusion detection, honeypot systems, and autonomous defense architectures. Section III presents the system architecture and agent-inspired pipeline design. Section IV details the hybrid risk scoring model. Section V describes the implementation. Section VI presents experimental evaluation and results. Section VII discusses limitations and future directions. Section VIII concludes the paper.


II. RELATED WORK

A. Signature-Based Intrusion Detection Systems

Snort [1] and Suricata [2] are the most widely deployed open-source network IDS platforms. Snort uses a rule-based engine that matches packet payloads against a database of known attack signatures, processing packets in a single-threaded pipeline [1]. Suricata extends this model with multi-threaded packet processing and protocol-aware detection [2]. Both systems output alerts in standardized formats but provide no automated response mechanism. When a threat is detected, an administrator must manually interpret the alert, correlate it with other events, assess the risk, configure a firewall rule, and verify its effectiveness. This manual process typically requires 15 to 30 minutes even for experienced operators [6], during which the attacker may have already achieved their objective.

Research by Alahmadi et al. [7] demonstrated that security analysts in enterprise environments investigate only 4-8% of generated IDS alerts, leading to significant threat coverage gaps. This alert fatigue problem motivates the need for autonomous systems that can act on detected threats without human intervention.

Unlike Snort and Suricata, our system implements a complete closed-loop that autonomously progresses from detection through scoring, decision-making, firewall enforcement, and honeypot deployment, achieving a mean end-to-end latency of approximately 127 ms.

B. Honeypot Systems

Cowrie [8] is a medium-interaction SSH honeypot that simulates a Linux shell environment, accepting any credentials and logging all commands typed by attackers. T-Pot [9] combines multiple honeypot services (Cowrie, Dionaea, Conpot, and others) into a single deployment with an Elasticsearch-Kibana visualization stack. Both systems provide valuable threat intelligence but require static pre-deployment and manual configuration. They have no mechanism for automatically redirecting suspicious traffic to honeypot services based on real-time threat assessment.

Vetterl and Clayton [17] analyzed attacker behavior in SSH honeypots and found that 72% of sessions last less than 60 seconds, highlighting the importance of immediate redirection to maximize intelligence capture. Our flow-aware dynamic deployment ensures that attackers are redirected to honeypots within milliseconds of detection, before they can disconnect.

Our system uniquely implements flow-aware dynamic honeypot deployment, where the Enforcement stage creates nftables NAT rules that transparently redirect specific attacker-to-victim traffic flows to appropriate honeypot services based on protocol analysis.

C. Machine Learning for Intrusion Detection

Numerous studies have applied supervised and unsupervised machine learning to network intrusion detection. Isolation Forest [3], an unsupervised anomaly detection algorithm based on random partitioning, has been shown effective for network anomaly detection due to its ability to identify outliers without requiring labeled attack data. Hariri et al. [18] extended the Isolation Forest algorithm and demonstrated its effectiveness for high-dimensional anomaly detection problems.

Random Forest classifiers have achieved detection rates exceeding 99% on benchmark datasets such as NSL-KDD and CICIDS2017 [4][19], but require labeled attack data for training. Ahmad et al. [20] compared multiple ML models for intrusion detection and found that while supervised models achieve higher accuracy, unsupervised models offer significant advantages in deployment flexibility.

Vinayakumar et al. [12] applied deep learning (CNN, LSTM, and hybrid architectures) to intrusion detection, achieving 97.2% accuracy on CICIDS2017 but requiring GPU hardware and extensive training time not suitable for real-time commodity hardware deployment.

Our system integrates the Isolation Forest model into a real-time scoring pipeline where anomaly scores directly influence enforcement decisions, creating a practical bridge between ML prediction and network defense action.

D. Autonomous Defense Architectures

The Autonomous Attack Response Framework (AARF) [13] employs multi-agent dynamic games and reinforcement learning to generate optimal response strategies for multi-step attack chains. While theoretically rigorous, AARF operates in simulation environments and does not report response latency measurements on real network hardware.

AETHER [14] proposes an AI-driven deception ecosystem using generative models to create dynamic decoy assets for advanced persistent threat detection. This system targets cloud infrastructure environments and does not address commodity hardware deployment or wired plus wireless convergent monitoring.

The LLM Agent Honeypot project [15] demonstrates detection of autonomous AI-driven hacking agents through prompt injection traps and timing analysis. While addressing the emerging threat of AI-powered attacks, this system focuses exclusively on detection and does not provide enforcement or automated response capabilities.

E. WiFi Probe Request Monitoring

WiFi probe request tracking has been extensively studied in the context of user privacy [16][21]. Mobile devices broadcast 802.11 Probe Request frames containing MAC addresses and previously connected network names, enabling passive device tracking. Vanhoef and Piessens [21] demonstrated that MAC address randomization can be defeated through timing analysis and Information Element fingerprinting. Matte et al. [22] further showed that devices can be re-identified despite randomization by analyzing the temporal patterns of probe request emissions.

Our system repurposes probe request monitoring for defensive security, using an Atheros AR9271 adapter in monitor mode to detect unauthorized device presence and deauthentication attacks on the monitored network.


III. SYSTEM ARCHITECTURE

A. Overview

NTTH implements an agent-inspired modular autonomous pipeline consisting of five processing phases and four specialized processing stages connected through an asynchronous event bus. The system captures network traffic from both wired interfaces (using Scapy [23] packet capture) and wireless interfaces (using the AR9271 adapter in monitor mode), analyzes each packet through a hybrid detection engine, and autonomously enforces appropriate responses through Linux kernel firewall rules and honeypot redirections. All events are persisted to a PostgreSQL database and streamed via WebSocket to a cross-platform Flutter dashboard.

B. Agent-Inspired Pipeline Design

We adopt an agent-inspired architecture where each processing stage operates as an autonomous module that perceives inputs, reasons about appropriate responses, and takes actions within its domain. We use the term "agent-inspired" following the classical definition by Russell and Norvig [24], where an agent is an entity that perceives its environment through sensors and acts upon it through actuators. Our processing stages implement this perception-reasoning-action cycle within their respective domains.

Table II describes each processing stage using this framework.

TABLE II
PROCESSING STAGE DEFINITIONS

Stage               | Perceives              | Reasons                    | Acts
Threat Assessment    | Packet features        | IDS rules + ML scoring     | Publishes risk assessment
Decision Making      | Risk score + context   | Protocol-aware policy      | Selects enforcement action
Enforcement          | Action directive       | Rule dedup + flow analysis | Applies nftables rules
Reporting            | Event data             | Victim identification      | Persists + broadcasts to UI

C. Asynchronous Event Bus

The processing stages communicate through an asynchronous publish-subscribe event bus implemented using Python asyncio queues with a capacity of 5,000 events. This design pattern is inspired by reactive event-driven architectures [25] commonly used in microservice systems. The event bus supports four topic channels:

    1) device_seen: Published by the packet sniffer for every captured packet, consumed by the Threat Assessment stage.
    2) threat_detected: Published by the Threat Assessment stage when risk score exceeds the logging threshold (0.15), consumed by the Decision Making stage.
    3) enforcement_action: Published by the Decision Making stage, consumed by the Enforcement stage.
    4) report_event: Published by the Enforcement stage, consumed by the Reporting stage for database persistence and WebSocket broadcast.

This decoupled design provides fault isolation — if the Enforcement stage fails to apply a firewall rule (e.g., due to permission issues), the Reporting stage still receives the event and records it for administrative review.

D. Flow-Aware Dynamic Honeypot Deployment

Traditional honeypot deployments are static: services are configured before attacks occur and operate independently of the detection system [8][9]. NTTH introduces flow-aware dynamic honeypot deployment, where the Enforcement stage creates targeted NAT redirection rules for specific network flows.

When the Decision Making stage determines that a threat warrants honeypot redirection (risk score between 0.40 and 0.80), it specifies the exact flow parameters:

    Source IP: The attacker's IP address
    Destination IP: The victim's IP address
    Destination Port: The target service port
    Honeypot Port: 2222 for SSH-like traffic, 8888 for HTTP-like traffic

The Enforcement stage translates this into an nftables NAT prerouting rule:

    nft add rule ip nat prerouting ip saddr {attacker} ip daddr {victim} tcp dport {port} dnat to 127.0.0.1:{honeypot_port}

This design provides three advantages over static honeypot deployment:

    1) Selective deception: Only the attacker's malicious traffic is redirected. Their legitimate traffic to other destinations passes through normally.
    2) Dynamic activation: Honeypot redirection is created in response to detected threats, not pre-configured.
    3) Contextual correlation: Each honeypot session is linked to the original threat assessment through a redirect context registry, enabling rich forensic analysis that maps Docker NAT IPs back to real attacker addresses.

The redirect context registry stores the mapping between the real attacker IP and the Docker bridge IP (172.18.0.x), enabling the honeypot session logger to correctly attribute sessions to their original sources despite Docker's network address translation.


IV. HYBRID RISK SCORING MODEL

A. Feature Extraction

For each captured packet, the system extracts a feature vector consisting of 10 dimensions from the IP and TCP/UDP headers:

    x = [packet_length, destination_port, is_syn, is_ack, is_rst, is_fin, protocol_encoded, tcp_window_size, ttl, payload_entropy]

Table III describes each feature and its relevance to intrusion detection.

TABLE III
FEATURE VECTOR DESCRIPTION

#  | Feature          | Type    | Range         | IDS Relevance
1  | packet_length    | Integer | 20–65535      | Anomalous sizes indicate crafted packets [12]
2  | destination_port | Integer | 0–65535       | Known service ports vs. unusual ports [1]
3  | is_syn           | Binary  | 0 or 1        | SYN flood and port scan indicator [1][2]
4  | is_ack           | Binary  | 0 or 1        | Connection state tracking [2]
5  | is_rst           | Binary  | 0 or 1        | Connection rejection indicator [1]
6  | is_fin           | Binary  | 0 or 1        | Connection teardown tracking
7  | protocol_encoded | Float   | 0.0–3.0       | TCP(1), UDP(2), ICMP(3), other(0)
8  | tcp_window_size  | Integer | 0–65535       | OS fingerprinting, zero-window attacks [2]
9  | ttl              | Integer | 0–255         | Spoofing detection, OS identification [7]
10 | payload_entropy  | Float   | 0.0–8.0       | Encrypted vs. plaintext traffic [12]

The protocol field is encoded as: TCP = 1.0, UDP = 2.0, ICMP = 3.0, other = 0.0. Boolean flags are encoded as 1.0 (true) or 0.0 (false). Features 8-10 (tcp_window_size, ttl, payload_entropy) extend the base feature set to capture OS-level characteristics and payload properties that improve discrimination between normal and anomalous traffic.

B. Signature-Based Rule Engine

Three sliding-window detectors operate on a per-source-IP basis, implemented using in-memory deque data structures:

    1) Port Scan Detector: Tracks unique destination ports per source IP within a 15-second window. If the count exceeds 4 unique ports, the detector returns a score of 1.0 (port_scan). Below threshold, the score is proportional: min(unique_ports / 4, 0.8). This threshold was calibrated based on the observation that normal browsing typically accesses 1-2 ports (80/443) per destination [7], while nmap [26] default SYN scans probe 1,000 ports.

    2) SYN Flood Detector: Counts SYN packets per source IP within a 1-second window. If the rate exceeds 30 SYN packets per second, the score is 1.0 (syn_flood). Below threshold: min(rate / 30, 0.8). The threshold of 30 SYN/sec was selected as it exceeds typical web browsing SYN rates (1-5/sec) [27] while being low enough to detect low-volume SYN floods from resource-constrained devices such as mobile phones running Termux.

    3) Brute Force Detector: Counts connection attempts to authentication ports within a 120-second window. Authentication ports include standard services (22, 23, 3389, 5900) and honeypot ports (2222, 8888, 30022). If attempts exceed 3, the score is 1.0 (brute_force). The threshold of 3 attempts in 120 seconds was chosen because legitimate SSH users typically succeed within 1-2 attempts, while automated tools like Hydra [28] generate rapid sequential attempts.

The rule score is the maximum across all three detectors:

    rule_score = max(port_score, syn_score, brute_score)

C. Isolation Forest Anomaly Detector

We employ the Isolation Forest algorithm [3] for unsupervised anomaly detection. The model is configured with 200 estimator trees and a contamination parameter of 0.05, assuming that approximately 5% of baseline traffic may be anomalous.

Training Procedure: The model operates in two phases. During the buffer phase, the first 500 packets of normal traffic are accumulated as baseline training data. Once 500 samples are collected, the Isolation Forest trains on this baseline and the model is persisted to disk (isolation_forest.joblib) for reuse across system restarts.

Scoring: For each subsequent packet, the model's decision_function() returns a signed anomaly score. This score is normalized and mapped to the range [0.0, 1.0] where 1.0 indicates maximum anomaly (most isolated from baseline distribution).

Justification for Isolation Forest: We select Isolation Forest over supervised alternatives for three reasons: (1) it is unsupervised and does not require labeled attack data for training, enabling deployment on any network without prior attack datasets [3]; (2) it trains on baseline normal traffic and naturally adapts to each network's unique traffic patterns; (3) it operates with O(t * n * log(psi)) time complexity where t is the number of trees, n is the sample size, and psi is the sub-sampling size, making it suitable for real-time per-packet scoring [18].

D. Weighted Risk Score

The final risk score is a weighted combination of both detection methods:

    risk_score = w_rule * rule_score + w_ml * ml_score

where w_rule + w_ml = 1.0.

To determine optimal weights, we performed a grid search over weight configurations (w_rule, w_ml) in the set {(0.0, 1.0), (0.1, 0.9), ..., (1.0, 0.0)} with step size 0.1, evaluating each configuration on 2,050 labeled test packets (1,500 normal + 550 attack). The optimal configuration was found at w_rule = 0.6, w_ml = 0.4, achieving a maximum F1-score of 0.958. This configuration balances the reliability of signature-based detection (which contributes high confidence for known attack patterns) with the flexibility of ML-based anomaly detection (which captures novel patterns missed by rules). Pure rule-based scoring (w_ml = 0.0) achieves lower FP rate (2.7%) but misses novel attacks, reducing F1. Pure ML scoring (w_rule = 0.0) detects more broadly but suffers from higher FP rate (8.2%), which is unacceptable for automated enforcement.

E. Action Thresholds

Based on ROC curve analysis of the optimal weight configuration, we defined five action thresholds that map risk scores to enforcement responses:

    risk_score < 0.15:           Allow (normal traffic, no action)
    0.15 <= risk_score < 0.25:   Log (record for review, no enforcement)
    0.25 <= risk_score < 0.40:   Rate Limit (throttle source to 10 packets/second)
    0.40 <= risk_score < 0.80:   Honeypot (redirect to deception service)
    risk_score >= 0.80:          Block (drop all packets from source)

The threshold of 0.15 for logging was selected at the point on the ROC curve where the true positive rate exceeds 90% for any attack class. The block threshold of 0.80 was placed to ensure less than 1% probability of blocking legitimate traffic based on the false positive distribution of the test dataset. The honeypot range (0.40–0.80) was designed to capture the majority of confirmed threats while reserving complete blocking for the highest-confidence detections.

B. Hardware Requirements

The system is designed for deployment on commodity hardware, in contrast to cloud-based solutions [13][14]:

    Processing: Any x86_64 laptop or desktop (tested on Lenovo IdeaPad 5, Intel i5, 16 GB RAM)
    WiFi Adapter: Atheros AR9271 USB adapter (approximately $10 USD, supports monitor mode and packet injection via ath9k_htc kernel driver)
    Network: Any LAN environment (Ethernet or WiFi hotspot)
    Storage: 10 GB minimum for database and logs
    Memory: 4 GB RAM minimum

C. Deployment Architecture

The system operates in a Docker Compose environment with three containers:

    1) Backend Container: Runs with host network mode (network_mode: host) to capture real LAN traffic via Scapy [23]. Contains the FastAPI server, packet sniffer, all four processing stages, and the built-in HTTP honeypot (port 8888).
    2) Cowrie Container: SSH honeypot [8] accessible on port 30022 (externally mapped to internal port 2222), connected via Docker bridge network with a shared volume (/cowrie_logs/) for real-time log access.
    3) PostgreSQL Container: Persistent database storage on port 5432.

Host network mode is critical because Docker bridge networking would prevent the packet sniffer from capturing actual LAN traffic, as the sniffer would only see traffic destined for the container itself.

D. Wireless Monitoring Module

The Atheros AR9271 USB adapter is configured in monitor mode using standard Linux wireless tools [33]:

    sudo ip link set wlan1 down
    sudo iw dev wlan1 set type monitor
    sudo ip link set wlan1 up

Once in monitor mode, the adapter captures all 802.11 frames within range on the 2.4 GHz band, including:

    Probe Requests: Broadcast by mobile devices seeking known WiFi networks [16]. We extract the source MAC address, requested SSIDs, and Received Signal Strength Indicator (RSSI) for device presence tracking.
    Deauthentication Frames: Type 0 subtype 12 management frames [33]. A high volume of deauth frames (>10 per second targeting the same BSSID) indicates a deauthentication attack, commonly used as a precursor to evil twin attacks.
    Beacon Frames: Broadcast by access points. We compare beacon SSIDs and BSSIDs against a whitelist of authorized access points to detect rogue AP deployments.

E. Database Schema

The system maintains four primary tables in PostgreSQL:

    1) threat_events: Records every detected threat with 15 fields including source IP, destination IP and port, protocol, threat type classification, risk score (0.0–1.0), action taken, GeoIP data (country, city, ASN, latitude, longitude), and incident notes stored as JSON.
    2) firewall_rules: Tracks every nftables rule with type (block/rate_limit/redirect), target IP, nft handle for cleanup, activation and expiry timestamps, and active status boolean.
    3) honeypot_sessions: Records every honeypot interaction with session ID, real attacker IP (resolved from Docker NAT), honeypot type (ssh/http), commands captured as JSON array, credentials attempted (username/password), connection and disconnection timestamps, and duration in seconds.
    4) devices: Tracks discovered network devices with IP, MAC address, hostname, vendor (from OUI lookup), device type classification, current risk score, and last seen timestamp.

F. Cowrie Log Watcher

The Cowrie SSH honeypot [8] writes structured JSON events to a log file that is shared with the backend container via Docker volume. A background asyncio task (cowrie_watcher) continuously tails this log file, parsing events as they appear:

    cowrie.login.success / cowrie.login.failed: Triggers session creation with captured credentials.
    cowrie.command.input: Appends the captured command to the session's command list.
    cowrie.session.closed: Updates the session with disconnection timestamp and duration.

For each event, the session logger resolves the Docker NAT IP back to the real attacker IP using the redirect context registry, performs GeoIP enrichment [32], upserts the session to the database, and broadcasts a WebSocket update with risk score 0.98 (honeypot interactions are inherently high-risk as they indicate confirmed malicious intent).


VI. EXPERIMENTAL EVALUATION

A. Testbed Configuration

Experiments were conducted on a Lenovo IdeaPad 5 15IAL7 laptop running Ubuntu 22.04 LTS with an Intel Core i5-1240P processor, 16 GB RAM, and a 512 GB NVMe SSD. The system was connected to a local WiFi network (802.11n, 2.4 GHz) serving as the monitored LAN. An Atheros AR9271 USB adapter was connected for wireless monitoring.

Attack traffic was generated from a secondary device (Android smartphone running Termux [34]) on the same network, simulating an internal threat actor. The attacker device had IP address 10.142.204.88 and the defender (NTTH host) had IP address 10.142.204.241. All experiments were conducted on a private, isolated network segment with no external connectivity.

B. Dataset

We validate our system on two datasets:

    Dataset 1 — Real-World (NTTH-RW): 10,247 packets captured from the production LAN during controlled attack scenarios over 5 days. The dataset consists of 7,697 normal traffic packets (web browsing, video streaming, file transfers, DNS queries) and 2,550 attack packets across four categories: port scan (1,050), brute force (600), SYN flood (500), and HTTP probing (400). The dataset was manually labeled by the researchers based on the time windows of attack execution.

    Dataset 2 — CICIDS2017 Benchmark [4]: 2,830,743 labeled network flow records with 80+ extracted features. We extracted our 10 features from the dataset and evaluated our Isolation Forest [3] model for cross-validation against an established benchmark.

VII. DISCUSSION

A. Strengths

The experimental results demonstrate three key strengths of the NTTH architecture:

    1) Sub-second autonomous response: The mean end-to-end latency of 127 ms from packet capture to firewall enforcement represents a qualitative improvement of over 99.98% compared to manual SOC response processes averaging 15–30 minutes [6]. This latency is sufficiently low to contain threats before attackers can establish persistence or achieve lateral movement.

    2) Flow-aware selectivity: Unlike IP-based blocking that affects all traffic from a source, flow-aware redirection preserves legitimate traffic while selectively redirecting malicious flows to honeypots. This reduces collateral impact on network usability and enables intelligence gathering without disrupting normal operations.

    3) Commodity accessibility: The entire system operates on a standard laptop with a $10 USB WiFi adapter, making autonomous network defense accessible to organizations that cannot afford enterprise security appliances. This contrasts with cloud-based solutions [13][14] that require dedicated infrastructure.

B. Limitation Analysis

We acknowledge several limitations that represent opportunities for future work:

    1) Single-host deployment: The current architecture monitors from a single network vantage point. Distributed multi-sensor deployment with a centralized coordinator would improve coverage for larger enterprise networks.

    2) 2.4 GHz wireless limitation: The AR9271 adapter operates only on the 2.4 GHz band [33]. Monitoring 5 GHz and 6 GHz (Wi-Fi 6E) networks requires additional hardware such as adapters based on the MediaTek MT7612U chipset.

    3) Docker NAT translation: When external traffic passes through Docker NAT to reach the Cowrie honeypot, the original source IP is translated to the Docker bridge IP (172.18.0.1). We mitigate this through redirect context correlation, but a host-mode Cowrie deployment would eliminate this translation entirely.

    4) Feature vector simplicity: Our 10-dimensional feature vector captures per-packet characteristics but does not model temporal sequences or session-level features. Incorporating LSTM or Transformer-based temporal modeling [12] could improve detection of slow-and-low attacks that distribute malicious activity over extended periods.

    5) Static policy engine: The Decision Making stage uses deterministic threshold-based policies rather than adaptive learning. Integrating reinforcement learning [13] for response strategy optimization could improve long-term defense effectiveness through experience-based adaptation.

    6) False positive rate: Our system's FP rate (3.8%) is higher than Snort (2.1%) and Suricata (1.8%) due to the ML anomaly detector. For automated enforcement, minimizing FPs is critical to avoid disrupting legitimate traffic. The planned Feedback Agent with administrator false positive reporting will enable continuous threshold refinement.

C. Ethical Considerations

All experiments were conducted on a controlled private network with informed consent of all participants. WiFi probe request monitoring was passive and limited to the researcher's own network segment. No data was collected from or transmitted to external networks without authorization. The system is designed exclusively for defensive security research and authorized network administration. The attack tools (nmap [26], Hydra [28], hping3 [35]) were used solely for controlled testing and not directed at any external systems.


VIII. CONCLUSION AND FUTURE WORK

This paper presented NTTH, an agent-inspired autonomous network defense architecture that implements a complete closed-loop from packet capture to attacker command capture. Through an asynchronous event bus connecting four specialized processing stages, the system achieves a mean response latency of 127 ms while providing hybrid detection combining IDS rules with Isolation Forest [3] anomaly scoring, intelligent protocol-aware enforcement via nftables, and dynamic flow-aware honeypot deployment.

Experimental evaluation on 10,247 real-world packets and the CICIDS2017 benchmark [4] demonstrated 96.2% detection rate with 3.8% false positive rate. Ablation study validated the optimal hybrid weight configuration (w_rule = 0.6, w_ml = 0.4, F1 = 0.958). Comparative evaluation against Snort [1] and Suricata [2] confirmed that NTTH uniquely provides automated containment within 127 ms while maintaining comparable detection accuracy, filling the critical gap between threat detection and enforcement that has persisted in traditional IDS approaches.

Future work will focus on four directions: (1) implementing a feedback agent that adapts detection thresholds based on administrator false positive feedback and periodically retrains the ML model, enabling continuous system improvement; (2) exploring deep reinforcement learning for optimal response strategy selection, inspired by AARF [13] but deployed on real hardware; (3) extending to multi-sensor distributed deployment for monitoring larger network environments with a centralized event bus coordinator; and (4) integrating Large Language Model (LLM) powered honeypot responses [15] to increase attacker engagement duration and intelligence gathering quality.
