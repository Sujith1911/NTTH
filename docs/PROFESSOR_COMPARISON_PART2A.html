3.1 — Snort (Signature-Based Network IDS)

Paper Details

TitleSnort — Lightweight Intrusion Detection for Networks
AuthorMartin Roesch
PublishedUSENIX LISA '99, pp. 229–238, 1999
Linkhttps://www.usenix.org/legacy/publications/library/proceedings/lisa99/full_papers/roesch/roesch.pdf
Citations6,000+ (one of the most cited IDS papers)

What Snort Does

Snort is the original open-source network intrusion detection system. It captures packets from a network interface and matches their content against a database of known attack signatures (currently 30,000+ community rules). When a signature matches, Snort writes an alert to a log file. Snort 3.x introduced multi-threaded packet processing, but the core model remains the same: capture, match, alert.

What Snort Does Not Do

  • No automated enforcement. Snort generates alert log entries. It does not add firewall rules, redirect traffic, or isolate devices. A human operator must read the alert, interpret it, and manually configure a firewall. Research by Alahmadi et al. (USENIX Security 2022) found that analysts investigate only 4–8% of generated alerts.
  • No ML anomaly detection. Snort relies entirely on pre-written signatures. If an attack does not match a known signature, Snort will not detect it. Zero-day attacks and behavioral anomalies are invisible.
  • No honeypot integration. Snort has no mechanism to redirect suspicious traffic to deception services. It cannot capture attacker commands or credentials.
  • No wireless monitoring. Snort processes wired packet captures only.
  • No real-time dashboard. Snort outputs text-based alerts. Visualization requires separate tools (Barnyard2, Snorby, etc.).

The gap we address: Snort detects threats but leaves a manual response gap. According to the IBM X-Force Threat Intelligence Index (2024), manual SOC response takes 15–30 minutes on average. 3.2 — Suricata (Multi-Threaded IDS/IPS)

Paper Details

TitleSuricata: Open Source IDS/IPS/NSM Engine
AuthorOpen Information Security Foundation (OISF)
PublishedOngoing open-source project, active since 2009
Linkhttps://suricata.io
Docshttps://docs.suricata.io/en/latest/

What Suricata Does

Suricata extends the Snort model with multi-threaded packet processing, protocol-aware detection, and rich JSON output (EVE logs). It can operate in IPS (inline) mode where it drops packets matching rules, but this mode requires specific network topology (inline bridge) and only performs binary drop/pass decisions. Suricata is widely regarded as the modern successor to Snort for high-throughput environments.

What Suricata Does Not Do

  • No graduated response. Suricata's IPS mode can only drop or pass a packet. It cannot rate-limit, redirect to a honeypot, or apply different actions based on risk level. The decision is binary.
  • No ML anomaly component. Like Snort, Suricata relies on signatures. Some research has added ML classifiers on top of Suricata's EVE output, but these are external additions, not part of Suricata itself.
  • No honeypot integration. Suricata cannot redirect traffic to deception services.
  • No attacker intelligence capture. Suricata does not record attacker commands or credentials.
  • Lower false positive rates (1.8%) but at the cost of missing novel attacks. Suricata's extensive signature database is highly precise for known patterns but blind to zero-day or behavioral anomalies.

3.3 — Fail2Ban (Log-Based Auto-Blocker)

Tool Details

TitleFail2Ban — Ban hosts that cause multiple authentication errors
TypeOpen-source log-monitoring and auto-ban tool
PublishedActive since 2004, continuously maintained
Linkhttps://github.com/fail2ban/fail2ban
Docshttps://www.fail2ban.org/wiki/index.php/Main_Page

What Fail2Ban Does

Fail2Ban monitors service log files (e.g., /var/log/auth.log, /var/log/apache2/error.log) using regex patterns. When it detects repeated failed login attempts from the same IP, it automatically adds an iptables/nftables ban rule for that IP for a configurable duration. It is extremely lightweight and effective at stopping brute-force SSH attacks.

Why This Comparison Matters

Important: This is the comparison a professor is most likely to challenge us on. "Why not just use Fail2Ban?" is a reasonable question, and we need to have a clear, honest answer. The answer is not that Fail2Ban is bad — it is that it solves a different, narrower problem.

What Fail2Ban Does Not Do

  • Log-based, not packet-based. Fail2Ban reads log files after a service has already processed the connection. It cannot detect port scans (no log entry is created for a SYN probe to a closed port). It cannot detect SYN floods (these happen at the network layer, below application logs). It can only react to events that produce log output.
  • No ML component. Fail2Ban uses regex pattern matching. It cannot learn normal traffic baselines or detect behavioral anomalies.
  • Binary response only. Fail2Ban either bans the IP entirely or does nothing. There is no graduated response — no rate-limiting, no honeypot redirection, no risk scoring.
  • No attacker intelligence. Fail2Ban blocks the IP and moves on. It does not redirect attackers to deception services or capture their commands and credentials.
  • No response latency measurement. Fail2Ban's response time depends on log file write latency, regex evaluation, and iptables execution. This latency is not measured or reported in any publication.
  • Single-service focus. Each Fail2Ban "jail" monitors one log file for one service. It does not correlate activity across services or provide a unified risk score.

Additionally, NTTH redirects attackers to honeypots for intelligence gathering instead of simply blocking them.

3.4 — OSSEC / Wazuh (Host-Based IDS)

Tool Details

TitleOSSEC — Open Source Host-based Intrusion Detection System
TypeHost-based IDS (HIDS) with active response
PublishedActive since 2004; Wazuh is the modern maintained fork
OSSEC Linkhttps://www.ossec.net/
Wazuh Linkhttps://wazuh.com/
Wazuh Docshttps://documentation.wazuh.com/current/index.html

What OSSEC/Wazuh Does

OSSEC is a host-based intrusion detection system that monitors individual hosts rather than network traffic. It performs log analysis, file integrity monitoring (FIM), rootkit detection, and policy compliance checking. Wazuh, its modern fork, adds an API, Elastic Stack integration, and centralized management. OSSEC includes an "active response" feature that can execute scripts (e.g., add an iptables rule) when threats are detected.

What OSSEC/Wazuh Does Not Do

  • Host-based, not network-based. OSSEC monitors files, logs, and system calls on individual machines. It does not capture network packets or analyze traffic patterns across the network. A port scan against the network would not be detected unless a service logs the connection attempt.
  • Active response is script-based with higher latency. OSSEC's active response executes shell scripts, which typically take 1–10 seconds. This is much slower than kernel-level nftables rule application.
  • No ML anomaly detection. OSSEC uses predefined decoder rules and log correlation. It does not incorporate machine learning models.
  • No honeypot integration. OSSEC cannot redirect traffic to deception services.
  • No per-flow traffic control. OSSEC's active response blocks or unbans IPs, but cannot selectively redirect specific flows.
  • Different attack surface. OSSEC detects host-level compromise (file changes, rootkits). NTTH detects network-level attacks (scans, floods, brute force) before they reach the host.

Complementary rather than competitive: OSSEC/Wazuh and NTTH actually address different layers. OSSEC monitors what happens inside a host. NTTH monitors what happens on the network before traffic reaches any host. In a complete security architecture, both would be valuable. Our contribution is the network-level autonomous response layer, which OSSEC does not provide.
