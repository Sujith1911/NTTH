ABSTRACT

The increasing sophistication of automated cyber attacks demands network defense systems capable of autonomous detection, decision-making, and enforcement without human intervention. Traditional Intrusion Detection Systems (IDS) such as Snort and Suricata generate alerts but lack automated response capabilities, leaving a critical gap between threat detection and containment that typically spans 15 to 30 minutes.

This project presents NTTH (No Time To Hack), an agent-inspired autonomous network defense architecture that implements a complete closed-loop pipeline from packet capture through threat scoring, firewall enforcement, and attacker intelligence gathering. The system employs a hybrid risk scoring model combining signature-based IDS rules (weighted 0.6) with an unsupervised Isolation Forest anomaly detector (weighted 0.4). Upon threat detection, the system autonomously applies Linux kernel-level nftables firewall rules and performs flow-aware dynamic redirection of specific attacker-victim-port flows to SSH (Cowrie) and HTTP honeypots, capturing attacker commands, credentials, and behavioral patterns in real-time.

The system architecture follows an agent-inspired modular pipeline with four autonomous processing stages — Threat Assessment, Decision Making, Enforcement, and Reporting — coordinated through an asynchronous event bus. A real-time Flutter dashboard (Web and Android) provides live visualization of threats, firewall rules, honeypot sessions, and network topology via persistent WebSocket connections. The system additionally integrates wireless monitoring using a commodity Atheros AR9271 USB WiFi adapter in monitor mode for probe request analysis and deauthentication attack detection.

The entire system is deployable on commodity hardware (Ubuntu laptop + $10 USB adapter), making enterprise-grade autonomous defense accessible for small-scale network environments. Experimental validation and comparative evaluation against Snort and Suricata are currently in progress.

Keywords: Intrusion Detection System, Honeypot, Autonomous Network Defense, Anomaly Detection, Isolation Forest, Firewall Automation, Agent-Inspired Architecture, WiFi Monitoring, Flutter Dashboard

CHAPTER 1: INTRODUCTION

1.1 Background

The rapid proliferation of networked devices and the internet has fundamentally transformed how organizations operate, but it has also expanded the attack surface available to malicious actors. Cyber attacks are increasing in both frequency and sophistication, with automated scanning tools, botnets, and AI-driven attack agents capable of exploiting network vulnerabilities at machine speed. The IBM X-Force Threat Intelligence Index 2024 reports that the average time from initial network compromise to lateral movement has decreased to under two hours, while the volume of automated attacks continues to grow exponentially.

Traditional network defense strategies rely on human-operated Security Operations Centers (SOCs) where analysts manually investigate alerts generated by Intrusion Detection Systems (IDS), assess severity, and configure firewall rules. This manual process typically requires 15 to 30 minutes per incident — a timeframe that is increasingly insufficient against automated attacks. Research has shown that the average enterprise SOC receives over 10,000 alerts per day, with analysts able to investigate only 4–8% of them, leading to significant threat coverage gaps known as "alert fatigue."

Existing IDS platforms such as Snort and Suricata excel at signature-based detection of known attack patterns but suffer from two critical limitations: they generate alerts without enforcement action, and they cannot detect novel attack patterns lacking known signatures. Honeypot systems like Cowrie and T-Pot offer attacker intelligence capture but require static manual deployment independent of detection systems.

1.2 Problem Statement

"Existing Intrusion Detection Systems detect network threats but cannot autonomously respond, leaving a critical 15–30 minute gap between detection and containment during which attackers achieve their objectives."

1.3 Motivation

The motivation for this project stems from the observation that while individual components of network defense — detection, ML scoring, firewalling, honeypots — exist in isolation, no existing system integrates all of them into a single closed-loop autonomous pipeline that operates on commodity hardware accessible to small organizations and educational institutions. The need for a system that can detect, decide, and enforce in sub-second latency without human intervention is both academically interesting and practically urgent.

1.4 Objectives

  1. To design and implement an agent-inspired autonomous pipeline that detects, scores, decides, and enforces network defense actions with sub-second response latency, eliminating the need for manual intervention.
  2. To develop a hybrid risk scoring model combining signature-based IDS rules with unsupervised Isolation Forest anomaly detection, validated through systematic ablation study.
  3. To implement flow-aware dynamic honeypot deployment that transparently redirects attacker flows to SSH (Cowrie) and HTTP deception services.
  4. To integrate wireless threat monitoring using a commodity Atheros AR9271 USB WiFi adapter in monitor mode.
  5. To develop a real-time cross-platform dashboard (Flutter Web + Android) with WebSocket live updates.

CHAPTER 2: LITERATURE REVIEW

2.1 Signature-Based Intrusion Detection Systems

Snort, created by Martin Roesch in 1999, is the most widely deployed open-source network IDS. It uses a rule-based engine that matches packet payloads against a database of known attack signatures, processing packets in a single-threaded pipeline. Suricata, developed by the Open Information Security Foundation (OISF), extends this model with multi-threaded packet processing, protocol-aware detection, and file extraction capabilities.

Both systems output alerts in standardized formats but provide no automated response mechanism. When a threat is detected, an administrator must manually interpret the alert, correlate it with other events, assess the risk, configure a firewall rule, and verify its effectiveness. Research by Alahmadi et al. demonstrated that security analysts in enterprise environments investigate only 4–8% of generated IDS alerts, leading to significant threat coverage gaps.

2.2 Honeypot Systems

Cowrie is a medium-interaction SSH honeypot that simulates a Linux shell environment, accepting any credentials and logging all commands typed by attackers. T-Pot combines multiple honeypot services (Cowrie, Dionaea, Conpot) into a single deployment with an Elasticsearch-Kibana visualization stack. Both systems provide valuable threat intelligence but require static pre-deployment and manual configuration. They have no mechanism for automatically redirecting suspicious traffic to honeypot services based on real-time threat assessment.

Vetterl and Clayton analyzed attacker behavior in SSH honeypots and found that 72% of sessions last less than 60 seconds, highlighting the importance of immediate redirection to maximize intelligence capture.

2.3 Machine Learning for Intrusion Detection

Isolation Forest, proposed by Liu, Ting, and Zhou (2008), is an unsupervised anomaly detection algorithm based on random partitioning. It identifies outliers by measuring how easily data points can be "isolated" through random axis-aligned cuts. Points that require fewer cuts to isolate are considered more anomalous. The algorithm has been shown effective for network anomaly detection due to its ability to identify outliers without requiring labeled attack data.

Random Forest classifiers have achieved detection rates exceeding 99% on benchmark datasets such as CICIDS2017, but require labeled attack data for training. Vinayakumar et al. applied deep learning approaches (CNN, LSTM) to intrusion detection, achieving 97.2% accuracy but requiring GPU hardware not suitable for commodity deployment.

2.4 Autonomous Defense Architectures

The Autonomous Attack Response Framework (AARF) employs multi-agent dynamic games and reinforcement learning for optimal response strategy selection. However, AARF operates in simulation and does not report real-world latency measurements.

AETHER proposes an AI-driven deception ecosystem using generative models to create dynamic decoy assets. This system targets cloud infrastructure and does not address commodity hardware.

The LLM Agent Honeypot project demonstrates detection of autonomous AI-driven hacking agents through prompt injection traps. It focuses on detection only without enforcement.

2.5 WiFi Probe Request Monitoring

Mobile devices broadcast 802.11 Probe Request frames seeking known WiFi networks. Freudiger demonstrated that these broadcasts reveal device presence and historical network associations. Vanhoef and Piessens showed that MAC address randomization can be defeated through timing analysis. Our system repurposes this monitoring for defensive security using the AR9271 adapter.

Each stage perceives inputs from the event bus, reasons about appropriate responses using its specialized logic, and takes actions within its domain.

3.5 Flow-Aware Dynamic Honeypot Deployment

Unlike static honeypots that are deployed before attacks occur, NTTH creates targeted NAT redirection rules for specific network flows in real-time. When the Decision Agent determines a honeypot response, the Enforcement Agent creates an nftables DNAT rule:

nft add rule ip nat prerouting ip saddr {attacker} ip daddr {victim} tcp dport {port} dnat to 127.0.0.1:{honeypot_port}

This provides three key advantages:

  1. Selective deception: Only the attacker's malicious traffic is redirected; legitimate traffic passes normally.
  2. Dynamic activation: Honeypot rules are created in response to detected threats, not pre-configured.
  3. Contextual correlation: Each honeypot session is linked to the original threat assessment through a redirect context registry.

4.3.2 Packet Sniffer (packet_sniffer.py)

The packet sniffer uses Scapy's AsyncSniffer to capture all IP packets on the network interface. It runs in host network mode to access real LAN traffic:

async def start_sniffer():
    from scapy.all import AsyncSniffer
    
    def _packet_callback(pkt):
        features = extract_features(pkt)
        if not features:
            return
        # Update live stats and device registry
        update_live_stats(features)
        registry_update(features)
        # Publish to event bus (thread-safe)

        )

        prn=_packet_callback,
        store=False,
        filter="ip",
        iface=settings.network_interface
    )
    sniffer.start()

    if src_ip == settings.gateway_ip:

4.4.4 Event Bus Core

_queue = asyncio.Queue(maxsize=5000)
_subscribers = defaultdict(list)

def subscribe(topic, handler):
    _subscribers[topic].append(handler)

async def publish(topic, payload):
    _queue.put_nowait({"topic": topic, "payload": payload})

async def _dispatch_loop():
    while True:
        event = await _queue.get()
        for handler in _subscribers[event["topic"]]:
            try:
                await handler(event["payload"])
            except Exception as exc:
                log.error("handler_error", error=str(exc))
        _queue.task_done()

4.4.5 Risk Calculator

def calculate(rule_score: float, ml_score: float) -> float:
    risk = THRESHOLDS.rule_weight * rule_score \
         + THRESHOLDS.ml_weight * ml_score
    return round(min(max(risk, 0.0), 1.0), 4)

def determine_action(risk_score: float) -> str:
    if risk_score >= THRESHOLDS.block_threshold:     return "block"
    if risk_score >= THRESHOLDS.honeypot_threshold:  return "honeypot"
    if risk_score >= THRESHOLDS.rate_limit_threshold: return "rate_limit"
    if risk_score >= THRESHOLDS.log_threshold:        return "log"
    return "allow"

4.5 Firewall and Honeypot Integration

4.5.1 nftables Manager

The NFTManager class wraps Linux nftables commands via subprocess calls, supporting three rule types: block (drop), rate_limit (limit rate 10/second), and redirect (DNAT to honeypot). Each rule is assigned a TTL (default 1 hour) and tracked in the database for auto-expiry.

4.5.2 Cowrie SSH Honeypot

Cowrie runs as a Docker container (image: cowrie/cowrie:latest) on port 30022 externally mapped to port 2222 internally. It simulates a Linux shell environment, logging all attacker activity to a JSON log file shared via Docker volume. The backend's cowrie_watcher continuously tails this log, parsing login events, commands, and session closures.

CHAPTER 7: CONCLUSION AND FUTURE WORK

7.1 Summary of Achievements

This project has successfully designed and implemented NTTH, an agent-inspired autonomous network defense architecture. The key achievements to date include:

  1. Complete closed-loop pipeline: A four-stage agent-inspired pipeline (Threat → Decision → Enforcement → Reporting) coordinated through an asynchronous event bus processes packets from capture to enforcement autonomously.
  2. Hybrid risk scoring: A weighted combination of signature-based IDS rules (3 detectors) and Isolation Forest anomaly detection provides both known-attack reliability and novel-attack flexibility.
  3. Real kernel enforcement: Linux nftables firewall rules are applied automatically, providing block, rate-limit, and honeypot redirect capabilities at the kernel level.
  4. Flow-aware honeypot deployment: Dynamic NAT redirection creates targeted honeypot sessions linked to original threat assessments through a redirect context registry.
  5. Full-featured dashboard: A Flutter Web + Android app with 9 screens provides real-time visualization via persistent WebSocket connections.
  6. Production deployment: The system runs in Docker Compose with PostgreSQL, Cowrie SSH honeypot, and the backend server, demonstrating real-world operability.
  7. Verified with real attacks: The system has been tested with nmap port scans, SSH brute force attacks, SYN floods, and HTTP probing, successfully detecting and containing all attack types.

7.2 Limitations

  • Single-host deployment: Currently monitors from a single network vantage point.
  • 2.4 GHz only: The AR9271 adapter operates only on 2.4 GHz band.
  • Docker NAT: Requires redirect context correlation to map Docker NAT IPs to real attackers.
  • Feature vector simplicity: 10-dimensional per-packet features do not model temporal sequences.
  • Static policy engine: Decision Agent uses deterministic thresholds, not adaptive learning.

7.3 Future Work

  1. Feedback Agent: Implement an adaptive threshold tuning mechanism based on administrator false positive feedback, transforming the system from agent-inspired to fully adaptive.
  2. AR9271 wireless monitoring: Complete the integration of monitor mode capture for WiFi probe requests, deauthentication detection, and rogue AP identification.
  3. Reinforcement learning: Explore RL-based optimal response strategy selection for the Decision Agent.
  4. LLM honeypot responses: Integrate Large Language Model powered responses to increase attacker engagement duration.
  5. Distributed deployment: Extend to multi-sensor architecture with centralized event bus coordination for enterprise networks.
